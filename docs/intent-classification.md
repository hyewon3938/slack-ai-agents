# 한국어 의도 분류 개선 기록

Slack AI 에이전트에서 한국어 메시지의 의도(잡담 vs 작업 요청)를 분류하는 시스템을 개선한 과정.

## 배경

사용자가 Slack에서 보내는 메시지는 크게 두 종류다:
- **잡담**: "고마워", "화이팅", "루틴 매일 지키기 힘드네~"
- **작업 요청**: "오늘 일정 보여줘", "루틴 추가해줘", "실키토마토 손질 끝냈어"

문제는 이 둘을 정확히 구분하는 게 한국어에서 특히 어렵다는 것이었다.

---

## 한국어 특유의 어려움

### 1. ㄹ-불규칙 활용

한국어 ㄹ-불규칙 동사는 특정 어미 앞에서 ㄹ이 탈락한다.

```
"힘들다" → "힘들어" (ㄹ 유지)
         → "힘드네" (ㄹ 탈락)
         → "힘든"   (ㄹ 탈락)
```

키워드 목록에 `"힘들"`만 있으면 `"힘드네"`를 감지하지 못한다. 같은 단어인데 형태가 달라지는 것이다.

**해결:** 활용형 변이를 키워드에 추가
```typescript
// 이전: '힘들' 하나만
// 이후: 활용형 모두 커버
CASUAL_OVERRIDES = ['힘내', '힘들', '힘드', '힘든']
```

### 2. 자동사/타동사 쌍

한국어는 같은 의미라도 자동사/타동사 형태가 다르다.

```
"끝나다" (자동사) → "끝났어" (저절로 끝남)
"끝내다" (타동사) → "끝냈어" (내가 끝냄)
```

"실키토마토 손질하는 것도 끝냈어"에서 `"끝냈"`은 `"끝났"`과 다른 단어다. 키워드에 `"끝났"`만 있으면 `"끝냈어"`를 잡담으로 오분류한다.

**해결:** 자동사/타동사 쌍 모두 등록
```typescript
ACTION_KEYWORDS = ['끝났', '끝냈', '끝내', '마쳤', '했어', '했다']
```

### 3. 감정 표현 + 액션 키워드 혼재

한국어에서 감정 표현과 액션 키워드가 자연스럽게 섞인다.

```
"내일 해보고 조정해봐야지"
  → "내일" = 액션 키워드 (날짜)
  → "해봐야지" = 감정 표현 (다짐)
  → 의도: 잡담 (혼잣말)
```

```
"나 루틴 되게 많지 ㅋㅋㅋ"
  → "루틴" = 액션 키워드
  → "ㅋㅋ" = 감정 표현
  → 의도: 잡담 (감탄)
```

이런 경우 키워드만으로는 판단이 불가능하다.

---

## 개선 전: LLM에 의존하는 분류

```
모든 메시지
  ↓
키워드 있음? → LLM에게 "이거 잡담이야 요청이야?" 질문 (2\~3초)
키워드 없음? → 잡담
```

**문제점:**
1. **느림** — 모든 액션 키워드 포함 메시지에 LLM 호출 (2\~3초)
2. **부정확** — LLM이 "내일 해보고 조정해봐야지"를 액션으로 잘못 분류
3. **비용** — 잡담 하나에도 LLM API 비용 발생

실제 사례:
- "루틴 매일 지키기 힘드네~" → LLM이 "action"으로 분류 → 체크리스트 반환 (잘못)
- "오늘 고마워" → LLM이 "action"으로 분류 → 에이전트 루프 진입 (잘못)

---

## 개선 후: 하이브리드 4-case 분류

```
사용자 메시지
  ↓
메시지 길이 > 80자? → 즉시 액션 (0ms)
  ↓
키워드 분석:
  ├─ 감정만 매칭 ("ㅋㅋ", "화이팅")      → 즉시 잡담 (0ms)
  ├─ 액션만 매칭 ("일정 보여줘")         → 즉시 액션 (0ms)
  ├─ 둘 다 없음 ("음...")              → 즉시 잡담 (0ms)
  └─ 둘 다 매칭 ("내일 화이팅")         → LLM 판별 (2\~3초)
```

**핵심 통찰:** LLM이 필요한 건 "둘 다 매칭"되는 진짜 애매한 경우뿐이다. 나머지는 키워드로 확정할 수 있다.

### 4가지 분기 상세

| Case | 조건 | 판정 | LLM | 소요 | 비율 |
|------|------|------|-----|------|------|
| 1 | 80자 초과 | 액션 | 없음 | 0ms | \~5% |
| 2 | 감정만 매칭 | 잡담 | 없음 | 0ms | \~30% |
| 3 | 액션만 매칭 | 액션 | 없음 | 0ms | \~50% |
| 4 | 둘 다 없음 | 잡담 | 없음 | 0ms | \~10% |
| 5 | 둘 다 매칭 | LLM 판별 | 1회 | 2\~3초 | \~5% |

**\~95%의 메시지가 LLM 없이 0ms에 분류된다.**

## 키워드 설계

### ACTION_KEYWORDS (액션 키워드)

에이전트 루프로 보내야 하는 메시지를 감지하는 키워드.

```typescript
// 일정 에이전트
const ACTION_KEYWORDS = [
  // CRUD 동사
  '추가', '삭제', '빼', '변경', '수정', '넣어', '만들어', '바꿔', '옮겨',
  '없애', '지워', '완료', '취소',
  // 상태 변경 (자동사/타동사 모두)
  '진행중', '진행', '시작', '끝났', '끝냈', '끝내', '마쳤',
  '했어', '했다', '안했', '못했', '미뤄', '연기',
  // 조회 의도
  '일정', '할일', '보여', '알려', '조회', '목록', '백로그',
  // 날짜 (시간 의도 암시)
  '오늘', '내일', '모레', '이번주', '다음주', '언제',
];
```

### CASUAL_OVERRIDES (감정 키워드)

이 키워드가 있으면 액션 키워드가 없는 한 무조건 잡담으로 처리.

```typescript
const CASUAL_OVERRIDES = [
  // 응원/격려
  '화이팅', '파이팅', '해볼게', '할게', '잘할게',
  // 감사/인사
  '고마워', '수고', '잘 자', '알겠어', '그럴게', '응 알겠',
  // 웃음
  'ㅋㅋ', 'ㅎㅎ',
  // 다짐 (잡담성)
  '응원', '해봐야지', '해야지', '할 수 있겠지',
  // 감정 표현 (ㄹ-불규칙 활용형 포함)
  '힘내', '힘들', '힘드', '힘든',
  '잘하고 있', '괜찮', '어떡해', '어떻게 하지',
  '피곤', '귀찮', '지쳤', '싫어',
];
```

### 설계 원칙

1. **어간 기반 매칭**: `text.includes(keyword)` — 활용 어미에 무관하게 어간만 매칭
   - `"힘드"` → "힘드네", "힘드네요", "힘드니까" 모두 매칭
2. **자동사/타동사 쌍**: 같은 의미의 두 형태 모두 등록
   - `"끝났"` (자동사) + `"끝냈"` (타동사)
3. **활용형 분산 등록**: ㄹ-불규칙 등 형태 변이가 큰 것만 추가
   - `"힘들"` + `"힘드"` + `"힘든"` (3가지 활용형)
4. **과도한 확장 금지**: 모든 활용형을 등록하지 않음. 자주 쓰는 형태만.

## 루틴 에이전트의 추가 방어: 체크리스트 오분류 방지

루틴 에이전트는 "루틴"이라는 단어가 포함되면 체크리스트를 바로 보여주는 빠른 경로가 있다. 하지만 "나 루틴 많지 ㅋㅋ"도 "루틴"이 포함되어 있어 체크리스트로 오분류됐다.

```typescript
// 이전: "루틴" 포함 + CRUD 아님 → 체크리스트 (잘못)
// 이후: "루틴" 포함 + CRUD 아님 + 감정 키워드 없음 → 체크리스트
const detectChecklistTarget = (text: string): ChecklistTarget | null => {
  if (EXACT_KEYWORDS.has(trimmed)) return 'all';  // "루틴" 정확 매칭은 유지
  if (
    trimmed.includes('루틴') &&
    !CRUD_KEYWORDS.some((k) => trimmed.includes(k)) &&
    !ANALYTICS_KEYWORDS.some((k) => trimmed.includes(k)) &&
    !DATE_KEYWORDS.some((k) => trimmed.includes(k)) &&
    !CASUAL_OVERRIDES.some((p) => trimmed.includes(p))  // ← 감정 키워드 체크 추가
  ) { ... }
};
```

**결과:**

| 입력 | 이전 | 이후 |
|------|------|------|
| "루틴" (정확 매칭) | 체크리스트 | 체크리스트 |
| "오늘 루틴" | 체크리스트 | 체크리스트 |
| "나 루틴 많지 ㅋㅋ" | 체크리스트 (오류) | LLM 에이전트 → 잡담 응답 |
| "루틴 화이팅" | 체크리스트 (오류) | LLM 에이전트 → 잡담 응답 |
| "아침 루틴 보여줘" | 전체 체크리스트 | 아침만 필터링한 체크리스트 |

## 퍼지 매칭: 에이전트 루프의 유연성

키워드 분류를 통과해서 에이전트 루프에 도달하면, LLM이 자연어를 유연하게 이해한다.

```
사용자: "실키토마토 손질 끝냈어"
                ↓
키워드 분류: "끝냈" 매칭 → 액션 (0ms)
                ↓
에이전트 루프: LLM이 Notion에서 "실키 토마토 손질하기"를 퍼지 매칭으로 찾음
                ↓
상태를 done으로 변경 + 일정 목록 표시
```

키워드 분류는 "에이전트 루프에 보낼지 말지"만 결정한다. 실제 작업은 LLM이 처리하기 때문에, 일정 이름을 정확히 말하지 않아도 된다.

| 사례 | 원인 | 수정 |
|------|------|------|
| "루틴 매일 지키기 힘드네~" → 체크리스트 | `"힘드네"`가 CASUAL_OVERRIDES에 없음 (ㄹ-탈락) | `"힘드"` 추가 |

## Phase 3: 분류 시스템 전면 제거 — LLM 에이전트 루프로 통합

### 전환 계기

위의 키워드 기반 분류가 잘 동작했지만, 근본적인 한계가 드러났다.

**"완료하고 잘거야" 사건:** 사용자가 "아직 하루가 끝난 게 아니니까, 오늘 전부 완료하고 잘거야!"라고 보냈는데, 이 메시지가 액션으로 분류되어 에이전트가 실제로 모든 일정을 완료 처리해버렸다. "완료"가 ACTION_KEYWORDS에 매칭되고, "거야"가 CASUAL_OVERRIDES에 없었기 때문이다.

이 버그를 고치기 위해 `'거야', '해낼', '끝내고', '해치우'`를 CASUAL_OVERRIDES에 추가했지만, 이 순간 문제의 본질이 보였다:

> **새로운 표현이 나올 때마다 키워드를 추가하는 건, AI 초창기의 규칙 기반 시스템 실수를 반복하는 것이다.**

한국어 다짐/결의 표현은 무한히 많다: "해치울 거야", "해낼 거야", "끝장볼 거야", "마무리할 거야"... 이걸 하나하나 등록하는 건 결코 끝나지 않는 게임이다.

### 추가 문제: 혼합 메시지

이진 분류의 또 다른 한계 — 잡담과 액션이 한 문장에 섞인 경우:

```
"고마워, 내일 미팅 추가해줘"
```

이전 시스템에서는:
- "고마워" → CASUAL_OVERRIDE 매칭 → 잡담으로 분류 → 미팅 추가 무시!
- 또는 "추가" → ACTION_KEYWORD 매칭 → 액션으로 분류 → 고마워 무시!

어느 쪽이든 절반만 처리된다. 이진 분류의 구조적 한계.

### 해결: 분류 단계 자체를 제거

LLM은 이미 에이전트 루프에서 MCP 도구를 자율적으로 호출하며, 도구를 호출하지 않을 판단도 할 수 있다. 잡담인지 액션인지를 사전에 분류할 필요가 없다 — LLM이 알아서 판단하면 된다.

```
변경 전:
  메시지 → 키워드 분류 → 잡담이면 바로 응답, 액션이면 에이전트 루프

변경 후:
  메시지 → 빠른 경로 체크 → 해당 없으면 에이전트 루프 (LLM이 자율 판단)
```

| 체크리스트 빠른 경로 (detectChecklistTarget) | 마찬가지로 최적화 |
| CHECKLIST_SKIP_PATTERNS | CASUAL_OVERRIDES의 축소판. "안녕 루틴 잔소리꾼"이 체크리스트를 트리거하지 않도록 하는 게이트가드 (10개) |
| 시스템 프롬프트의 잡담 가이드 | 에이전트 루프의 유일한 잡담 방어선. "다짐 vs 명령" 구분 규칙 포함 |
| 환각 가드 (agent-loop.ts) | 도구 미호출 + 액션 동사 응답 → 1회 재시도. 잡담에는 무관 |

### 잡담 방어: 프롬프트 기반

분류 시스템 대신, 시스템 프롬프트에 잡담 판단 가이드를 포함:

```
## 잡담
- 일정/루틴과 무관한 가벼운 대화에는 도구 호출 없이 짧게 응답해.
- 다짐/결의/의지 표현은 잡담이야. 도구를 호출하지 마.
  잡담 예: "오늘 루틴 다 해치울 거야!", "루틴 열심히 해야지"
  명령 예: "루틴 추가해줘", "스트레칭 꺼줘"
  핵심 차이: "~할 거야/~하겠어"(나의 다짐) vs "~해줘/~처리해"(너에게 명령)
```

LLM이 이 규칙을 따라 잡담에는 도구 없이 응답하고, 명령에는 도구를 호출한다.

### 트레이드오프

**얻은 것:**
- CASUAL_OVERRIDES 키워드 땜질 완전 제거
- 혼합 메시지 자연 처리 ("고마워, 내일 미팅 추가해줘")
- 코드 \~500줄 삭제 (casual-chat.ts + 분류 로직)
- 새 표현 등장 시 코드 수정 불필요

**잃은 것:**
- 잡담에도 ack 메시지가 먼저 나감 ("잠깐만" → 1초 뒤 → "화이팅!")
- 잡담 응답 시간: 0\~2초 → 1\~3초 (MCP 도구 컨텍스트 로딩 포함)

**허용 이유:** 잡담 빈도가 낮고 (\~10%), 1\~3초 응답은 허용 범위. "완료하고 잘거야" 같은 오분류로 데이터가 파괴되는 것보다 훨씬 낫다.

## 전략 진화 요약

```
Phase 1: LLM 분류
  모든 메시지 → LLM에 "잡담? 액션?" 질문
  문제: 느림 (2\~3초), LLM도 오판

Phase 2: 하이브리드 키워드 분류
  키워드 → 4-case 분류 (95% 즉시, 5% LLM)
  문제: 새 표현마다 키워드 추가 (끝없는 땜질)

Phase 3: 분류 제거 → 에이전트 루프 통합 ← 현재
  빠른 경로 → 나머지 전부 에이전트 루프 (LLM 자율 판단)
  잡담 방어: 프롬프트 가이드 + 환각 가드
```

핵심 교훈: "분류"라는 별도 단계가 필요하다는 가정 자체가 틀렸다. LLM 에이전트는 이미 도구를 쓸지 말지 판단하는 능력이 있고, 이 판단에 잡담/액션 구분이 자연스럽게 포함된다.
