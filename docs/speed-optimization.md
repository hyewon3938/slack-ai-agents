# 속도 최적화 기록

Slack AI 에이전트의 응답 속도를 개선한 과정을 기록한 문서.

## 배경

Slack에서 자연어로 일정/루틴을 관리하는 AI 에이전트를 만들면서, 사용자 체감 속도가 가장 큰 과제였다. 모든 메시지가 LLM API를 거쳐야 했기 때문에 단순한 조회조차 5\~8초가 걸렸고, 잡담 한 마디에도 2\~3초씩 기다려야 했다.

## 핵심 병목: LLM API 호출

```
1회 LLM API 호출 ≈ 2~3초 (Gemini-2.5-flash 기준)
1회 MCP 도구 호출 ≈ 1~2초 (Notion API 경유)
1회 Notion SDK 직접 호출 ≈ 0.5~1초
```

LLM 호출 횟수가 곧 응답 시간이다. 최적화의 핵심은 **불필요한 LLM 호출을 제거**하는 것이었다.

---

## 개선 전: 모든 메시지가 LLM을 거치는 구조

```
사용자: "오늘 일정 보여줘"
  ↓
[1] LLM 의도 분류 (2~3초)    ← "이게 잡담인지 요청인지 판단"
  ↓
[2] LLM 에이전트 시작 (2~3초)  ← "Notion에서 일정을 가져와야겠다"
  ↓
[3] MCP 도구 호출 (1~2초)      ← Notion API로 일정 조회
  ↓
[4] LLM 응답 생성 (2~3초)      ← "오늘 일정 정리해서 알려줄게"
  ↓
총 소요: 7~11초
```

단순 조회에도 LLM이 3번 호출되고, MCP가 1번 호출됐다. "오늘 일정"처럼 매일 쓰는 기능에 이 시간은 치명적이었다.

---

## 개선 후: 3단계 빠른 경로 분기

```
사용자 메시지 수신
  ↓
[Gate 1] 잡담 감지 — 키워드 기반 즉시 판단 (0ms)
  ├─ "고마워", "ㅋㅋ" → 즉시 잡담 응답
  └─ 통과 →
[Gate 2] 조회 빠른 경로 — SDK 직접 호출 (~1초)
  ├─ "오늘 일정", "내일 뭐 있어" → Notion SDK 직접 조회 → 포맷팅 → 응답
  └─ 통과 →
[Gate 3] LLM 에이전트 루프 — 복잡한 요청만 (4~8초)
  └─ "이번주 일정 정리해줘", "미팅 추가해줘" → LLM + MCP
```

각 게이트에서 처리 가능하면 바로 응답하고 다음 단계로 넘기지 않는다.

---

## 시나리오별 속도 비교

| 시나리오 | 개선 전 | 개선 후 | 개선율 |
|---------|---------|---------|--------|
| **잡담** ("고마워", "ㅋㅋ") | 2\~3초 (LLM 분류) | **0ms** (키워드 즉시 판단) | - |
| **간단 조회** ("오늘 일정") | 7\~11초 (LLM 3회 + MCP) | **\~1초** (SDK 직접) | **7\~11배** |
| **일정 변경 후 목록 표시** | 7\~10초 (LLM 3회 + MCP + LLM 재조회) | **4\~6초** (LLM + MCP + SDK 직접 조회) | **\~2배** |
| **복잡한 요청** ("이번주 일정 정리") | 7\~11초 | 5\~8초 (분류 LLM 생략) | 소폭 개선 |

## 최적화 상세

### 1. 조회 빠른 경로 (Gate 2)

가장 큰 속도 개선. "오늘 일정", "내일 뭐 있어" 같은 단순 조회를 LLM 없이 처리한다.

**동작 방식:**
1. `detectSimpleQuery(text)` — 메시지 패턴 분석
   - 20자 이하 + 쓰기 키워드 없음 + 날짜 키워드(오늘/내일/모레) + 조회 의도
2. 매칭되면 `queryTodaySchedules()` — Notion SDK 직접 호출 (\~0.5초)
3. `buildQueryResponse()` — 기존 크론 포맷 재사용으로 응답 생성

**매칭되는 패턴:**
- "오늘 일정", "오늘 일정 보여줘", "내일 뭐 있어", "모레 일정", "일정 알려줘"

**매칭되지 않는 패턴 (LLM 에이전트로 위임):**
- "이번주 일정" (복잡한 날짜)
- "일정 추가해줘" (쓰기 작업)
- "백로그 보여줘" (특수 조회)

### 2. 잡담 즉시 판단 (Gate 1)

키워드 기반으로 LLM 호출 없이 잡담을 감지한다.

**4가지 분기:**

| 조건 | 결과 | LLM 호출 | 소요 |
|------|------|----------|------|
| casualOverride만 매칭 ("ㅋㅋ", "화이팅") | 즉시 잡담 | 없음 | 0ms |
| actionKeyword만 매칭 ("일정 추가") | 즉시 액션 | 없음 | 0ms |
| 둘 다 없음 ("음...") | 즉시 잡담 | 없음 | 0ms |
| 둘 다 매칭 ("내일 화이팅") | LLM 판별 | 1회 | 2\~3초 |

LLM을 호출하는 건 진짜 애매한 경우(\~5%)뿐이다. 나머지 95%는 0ms.

### 3. 변경 후 일정 표시 최적화

일정 추가/수정 후 해당 날짜 전체 일정을 보여주는 기능. LLM이 재조회하지 않고 코드 레벨에서 처리한다.

**동작 방식:**
1. `AgentLoopResult.toolNames` — 에이전트 루프에서 호출된 MCP 도구 이름 추적
2. `hasMutation()` — 변경 도구(API-post-page 등) 사용 여부 확인
3. 변경이 있으면 `extractMutationDate()` — 사용자 텍스트에서 대상 날짜 추출
4. `buildPostMutationList()` — Notion SDK로 해당 날짜 일정 조회 → 응답에 추가

기존: LLM이 변경 후 다시 API-post-search를 호출 (추가 LLM 라운드 2\~3초)
개선: SDK 직접 조회 (\~0.5초)로 LLM 라운드 1회 절약.

### 4. LLM 모델 선택

| 모델 | 가격 (입력/출력) | 속도 | 비고 |
|------|---------------|------|------|
| gemini-2.0-flash | 사용 불가 | - | 신규 사용자 차단됨 |
| gemini-2.5-flash | $0.15/$0.60 | 빠름 | **현재 사용 중** (GA) |
| gemini-3-flash-preview | $0.15/$0.60 | 빠름 | Preview, 불안정 가능 |

안정성과 속도를 고려해 GA 버전인 gemini-2.5-flash를 선택했다.

## Phase 2: 지연 ack + 도구 필터 캐싱

### 배경

의도 분류 로직을 제거하고 모든 비-빠른경로 메시지를 LLM 에이전트 루프로 통합한 후, 잡담("고마워")에도 항상 ack("잠깐만, 확인해볼게.")가 먼저 전송되는 문제가 생겼다.

```
기존 흐름:
  사용자: "고마워"
    → [즉시] ack: "잠깐만, 확인해볼게."  ← 불필요하게 어색
    → [~1초] LLM 응답: "응, 힘내."
```

잡담은 도구 호출 없이 1초 이내에 끝나는데, 매번 "잠깐만"이 먼저 나오면 이상하다.

### 해결: `runAgentLoopWithAck` — 지연 ack

ack 전송을 800ms 지연시킨다. LLM이 그 안에 응답하면 ack 생략.

```
개선 흐름 (빠른 응답):
  사용자: "고마워"
    → LLM 처리 시작 + 800ms 타이머 시작
    → [~1초] LLM 응답 완료 → 타이머 취소
    → 응답만 전송: "응, 힘내."

개선 흐름 (느린 응답):
  사용자: "미팅 추가해줘"
    → LLM 처리 시작 + 800ms 타이머 시작
    → [800ms] 타이머 발동 → ack: "잠깐만, 확인해볼게."
    → [4~6초] LLM 응답 완료
    → 응답 전송: "내일 미팅 추가했어."
```

**구현:** `src/shared/agent-loop.ts`의 `runAgentLoopWithAck`. 기존 `runAgentLoop`을 감싸는 래퍼 함수.

```typescript
const loopPromise = runAgentLoop(llmClient, userText, config);
const timer = setTimeout(() => { ackPromise = sendAck(); }, delayMs);
const result = await loopPromise;
clearTimeout(timer);
if (ackPromise) await ackPromise; // 순서 보장
```

### 도구 필터 캐싱

에이전트별 MCP 도구 필터링이 매 메시지마다 반복되던 것을 참조 동등성 캐싱으로 최적화.

```
기존: 매 메시지마다 getMCPTools() → filter() (배열 재생성)
개선: source === cachedSource이면 캐시 반환 (O(1))
      MCP 재연결 시 source 참조 변경 → 자동 무효화
```

**적용 파일:** `src/agents/schedule/tools.ts`, `src/agents/routine/tools.ts`

| 시나리오 | Phase 1 | Phase 2 | 체감 |
|---------|---------|---------|------|
| 잡담 ("고마워") | ack + 응답 (2개 메시지) | 응답만 (1개 메시지) | ack 어색함 제거 |
