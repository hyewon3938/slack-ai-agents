# 개발자 프로필 — AI 협업 관점의 분석

> 이 문서는 AI(Claude)와의 협업 과정에서 관찰된 개발자의 작업 스타일, 의사결정 패턴, 성향을 기록한다.
> 메타인지 도구이자, 프로젝트 설명 시 참고 자료로 활용.
> **마지막 업데이트:** 2026-03-10

---

## 1. 의사결정 스타일

### 실용적 미니멀리스트

과도한 설계를 경계하면서도 확장 가능한 구조를 놓치지 않는다.
"1단계만 먼저 적용하자"는 표현이 자주 등장하며, 필요해질 때 확장하는 점진적 접근을 선호한다.

대표 사례:
- 스마트 메모리 설계 시 벡터 검색, 맥락 기반 로딩 등 고급 기능을 인지하면서도 "개인용이면 1단계면 충분하다"고 판단
- v1 → v2 전환에서 "기능 추가로 보수하기보다 코어를 교체하는 근본 전환" 결정
- 의도 분류 시스템을 키워드 → LLM → 다시 제거하는 과정에서 "분류 자체가 불필요하다"는 근본적 질문에 도달

### 비교 기반 학습

새 기능을 도입할 때 먼저 유사 프로젝트를 연구하고, 차이점을 분석한 뒤, 자기 프로젝트에 맞게 재설계한다.
"오픈클로랑 내 프로젝트 차이점이 뭐야?"라는 질문이 전형적인 패턴.
남의 것을 그대로 가져오지 않고, 원리를 이해한 뒤 규모에 맞게 축소/변형한다.

### 엣지 케이스 선제 인식

구현 전에 실패 시나리오를 먼저 떠올린다.
스마트 메모리 설계를 논의할 때 기능 요청과 동시에 "이러면 중요 프롬프트가 소실될 위험이 있나?"를 바로 질문했다.
이 질문이 soft-delete + source 이중 구조라는 핵심 설계를 이끌어냄.

## 2. 지시 스타일

### 복합 지시 + 맥락 제공

하나의 메시지에 여러 요청을 묶되, 각각의 배경과 이유를 함께 전달한다.

예시:
> "일정 안에 메모 적어둘 수 있는 기능 추가해줘. 메모 저장할 때 줄바꿈 띄어쓰기 다 포함되도록 해줘. 마크다운으로 저장되면 더 좋고."

세 가지 요청(메모 기능 / 원문 보존 / 마크다운 지원)이 하나의 흐름으로 연결되어 있고, 우선순위도 암묵적으로 드러난다.

### 문제 → 원인 분석 → 해결 순서

"이거 고쳐줘"보다 "이거 이상한데, 원인 찾아봐"가 먼저 나온다.
KST 타임존 버그 발견 시에도 "3월 28일이 토요일인데 금요일이라고 나온다. 원인 찾아봐"로 시작.
원인을 이해한 뒤에 수정을 지시하는 패턴이 일관적이다.

### 프로세스 감각

커밋 타이밍, 브랜치 전략, PR 단위를 스스로 판단하고 제안한다.
"이거 보고나서 커밋할지 PR할지 정해봐"처럼 Claude에게 판단을 위임하기도 하지만, 기준을 가지고 있다.

## 3. 프로젝트 관리 성향

### 기록 중시

- GitHub Issues로 개발 계획을 체계적으로 관리
- `docs/project-history.md`에 설계 변화와 의사결정 과정을 상세 기록
- 코드 컨벤션 문서화, CLAUDE.md 유지
- 이 문서(developer-profile.md) 자체가 기록 중시 성향의 증거

### 구조화된 진행

1 Issue = 1 PR = 1 Feature 원칙을 지킨다.
Conventional Commits를 한글로 사용하며, 커밋 메시지가 변경의 "왜"를 설명한다.
브랜치 전략이 명확하고 (feature/xxx → main), 혼재를 싫어한다.

### 회고와 방향 전환

v1에서 v2로의 전환 과정이 대표적.
"잘된 점 → 드러난 한계 → 결론"의 구조화된 회고를 통해 근본 전환을 결정.
감정적 판단("이건 실패다")이 아닌 구조적 분석("JOIN 불가 → 크로스 분석 불가")으로 방향을 전환한다.

## 4. 기술적 성향

### UX 감각이 있는 백엔드 사고

주로 백엔드/시스템 설계를 하지만, 최종 사용자 경험에 민감하다.
- "루틴 체크리스트가 대화에 밀려서 올라가는 게 불편하다" → App Home 탭 도입
- 일정 표시 포맷을 세밀하게 설계 (카테고리 그룹, 상태 아이콘, 메모 └ 포맷)
- Block Kit 레이아웃에 대한 구체적인 기대치가 있음

### AI 시스템 이해도

LLM의 tool use 메커니즘, 프롬프트 엔지니어링, 에이전트 루프 패턴을 깊이 이해한다.
- "LLM이 요일을 머릿속으로 계산한다"는 문제를 정확히 진단
- 프롬프트 규칙의 강도 조절 ("절대 규칙"으로 격상)
- 오픈클로 아키텍처를 "시스템 프롬프트 + 도구 + 컨텍스트 주입"으로 정확히 분해

### 데이터 안전 우선

기능보다 데이터 안전을 우선한다.
- soft-delete를 자연스럽게 요구
- 마이그레이션 시 기존 데이터 호환성 확인
- SQL 도구에 DDL 차단, 위험 쿼리 방지 로직 설계

## 5. AI 협업 패턴

### 위임과 검증의 균형

- 탐색/조사는 적극 위임: "이거 원인 찾아봐", "오픈클로랑 비교해줘"
- 설계 결정은 함께 논의: "이렇게 하면 어때?", "이거 보고 커밋할지 정해봐"
- 최종 판단은 직접: "1단계만 먼저 하자", "히스토리에 남기자"

### 점진적 요구 확장

하나의 기능을 요청한 뒤, 결과를 보고 추가 요구를 붙인다.
이는 산만함이 아니라, 실제 동작을 보면서 빠진 부분을 발견하는 실용적 접근.

예시 흐름:
1. "메모 기능 추가해줘"
2. (결과 확인 후) "줄바꿈 보존되게 해줘. 마크다운도."
3. (배포 후) "아직도 요일이 틀리네. 다시 확인해봐."

### 메타 관점 유지

프로젝트를 만들면서 동시에 프로젝트에 대해 생각한다.
- 개발 히스토리 문서를 통해 과정 자체를 기록
- 다른 프로젝트(오픈클로)와 비교하며 자기 프로젝트의 위치를 파악
- 이 문서의 존재 자체가 "나는 어떤 개발자인가"에 대한 메타인지 시도

## 6. 개선 포인트

### 스코프 확장 경향

하나의 기능을 요청한 뒤 결과를 보면서 추가 요구를 붙이는 패턴이 반복된다.
개인 프로젝트에서는 유연한 탐색이지만, 팀 환경에서는 스코프 크리프로 이어질 수 있다.

예시 흐름:
1. "메모 기능 추가해줘"
2. "줄바꿈 보존되게 해줘"
3. "마크다운도"

각각은 합리적이지만, 처음부터 "마크다운 지원 메모 기능"으로 한 번에 정의했다면 구현 효율이 더 높았을 것.
**팁:** 기능 요청 시 "최종 상태"를 먼저 그려보고, 거기서 역산해서 요구사항을 정리하는 습관을 들이면 좋겠다.

### 검증 타이밍

첫 번째 KST 수정 후에도 요일이 여전히 틀렸던 사례가 있다.
코드 레벨 수정(formatDateShort)은 맞았지만, LLM 프롬프트 레벨(요일 추측)은 확인하지 않고 넘어갔다.
수정 후 "어떤 경로에서 이 값이 나오는지" 전체 흐름을 한번 더 추적하는 습관이 있으면 재작업을 줄일 수 있다.

**팁:** 버그 수정 시 "이 값이 사용자에게 도달하는 모든 경로"를 체크리스트로 나열해보면 누락을 방지할 수 있다.

### 하나의 메시지에 여러 주제

"이거 고쳐주고, 저것도 추가하고, 그리고 또..."식의 복합 요청이 잦다.
효율적이지만, 각 요청의 우선순위와 의존관계가 암묵적으로만 전달된다.
팀 협업에서는 이슈/티켓을 분리하는 것이 추적성과 리뷰 효율에 더 유리하다.

**팁:** 복합 요청 시 번호를 매기거나, "먼저 A, 그 다음 B" 같은 명시적 순서를 붙이면 더 명확해진다. (사실 이미 가끔 하고 있긴 함)

### 문서화 부담 관리

히스토리, 컨벤션, 프로필, CLAUDE.md 등 문서가 다양한데, 각 문서의 유지보수 비용이 쌓일 수 있다.
문서가 늘어날수록 "어디에 뭐가 적혀있는지" 관리하는 데 시간이 든다.

**팁:** 문서 간 역할을 명확히 구분하고, 정기적으로 "이 문서 아직 필요한가?" 리뷰하면 좋겠다.

### AI 탐색 위임과 직관 균형

버그 원인 조사, 아키텍처 비교 등 탐색 작업을 AI에게 적극 위임하는 것은 효율적이지만,
직접 코드를 따라가며 디버깅하는 경험이 줄어들면 코드베이스에 대한 직관이 약해질 수 있다.

**팁:** 중요한 버그는 가끔 직접 추적해보는 것도 좋다. "원인 찾아봐" 대신 "같이 원인 찾아보자"로 바꾸면 AI의 분석과 자신의 직관을 동시에 키울 수 있다.

## 7. 관찰 메모

> 이 섹션은 협업 중 눈에 띄는 포인트를 시간순으로 기록한다.

### 2026-03-08

- **v1 → v2 전환 결정의 속도**: "3D 프린터로 뗀석기를 만드는" 비유로 현재 구조의 한계를 표현. 감정적 미련 없이 근본 전환을 결정하는 것이 인상적.
- **오픈클로 연구**: 단순 호기심이 아닌, "내 프로젝트에 뭘 가져올 수 있나"라는 실용적 관점에서 접근. 원리를 이해한 뒤 규모에 맞게 축소 적용하는 패턴.
- **데이터 소실 질문**: 스마트 메모리 기능을 요청하면서 동시에 위험을 질문. 기능과 안전성을 동시에 사고하는 습관.
- **메타인지 도구 요청**: "나에 대한 분석 문서를 만들어달라"는 요청 자체가, 자기 개발 과정을 객관화하려는 의지. 단순히 코드를 짜는 것이 아니라, 자신의 작업 방식을 관찰하고 개선하려는 태도.

### 2026-03-10

- **작업 요약**: AI 개발 워크플로우 자동화 + 생활 맥락 인식 잔소리 시스템 + LLM 비용 최적화 + GitHub Actions CI/CD 구축 + README 포트폴리오 공개 + 백로그/내일일정 fast path 추가 + lint 정리 (25개+ 커밋)
- **성향**: 하루에 인프라(CI/CD), 기능(생활 맥락), 비용(LLM 최적화), 문서(README 포트폴리오), 성능(fast path)을 동시에 진행하는 전방위 작업 방식. 하루 마무리를 기능 개발이 아닌 "외부에 어떻게 보일까"로 마무리한 것이 인상적 — 개발자이면서 동시에 퍼블리셔 감각을 갖고 있다. CI/CD 도입 직후 lint 에러를 즉시 수정하는 빠른 피드백 루프도 눈에 띈다.
- **강점**: "Hooks + Skills + GitHub Actions + MCP"를 하나의 워크플로우로 묶는 시스템 사고. LLM 비용 최적화(하이브리드 모델)에 이어 fast path(LLM 바이패스)까지 성능·비용을 연속적으로 개선하는 일관된 최적화 사고방식. CI/CD가 lint 에러를 잡아내자 바로 fix PR을 올리는 것처럼, 자동화 도구의 피드백을 즉시 내면화하는 능력.
- **개선**: GitHub Actions를 한 번 만들고 삭제한 뒤 다시 추가하는 과정이 있었음(`004d475` 삭제). 빠른 실험-검증 패턴은 좋지만, 초기 설계 단계에서 "CI/CD에서 뭘 검증해야 하는가"를 먼저 정의했다면 왕복을 줄일 수 있었다.
- **AI 협업 팁**: fast path처럼 LLM을 바이패스하는 최적화를 설계할 때, "어떤 패턴이 fast path가 될 자격이 있는가"를 AI와 먼저 정의하면 추후 패턴 추가가 더 일관성 있게 된다. 정규식 기반 매칭 조건을 AI가 제안하고 사람이 검토하는 분업이 효과적.
