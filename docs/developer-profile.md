# 개발자 프로필 — AI 협업 관점의 분석

> 이 문서는 AI(Claude)와의 협업 과정에서 관찰된 개발자의 작업 스타일, 의사결정 패턴, 성향을 기록한다.
> 메타인지 도구이자, 프로젝트 설명 시 참고 자료로 활용.
> **마지막 업데이트:** 2026-03-12

---

## 1. 개발 철학

### 실용적 미니멀리즘

과도한 설계를 경계하면서도 확장 가능한 구조를 놓치지 않는다.
"1단계만 먼저 적용하자"는 접근이 일관적이며, 필요해질 때 확장하는 점진적 설계를 선호한다.

- 스마트 메모리 설계 시 벡터 검색 등 고급 기능을 인지하면서도 "개인용이면 1단계면 충분하다"고 판단
- 의도 분류 시스템을 키워드 → LLM → 다시 제거하는 과정에서 "분류 자체가 불필요하다"는 근본적 질문에 도달
- v1 → v2 → v3로 이어진 아키텍처 전환에서 매번 현재 구조의 한계를 구조적으로 분석하고 근본 전환을 결정

### 시스템 구축 지향

문서화, 컨벤션, 워크플로우 자동화를 개별 작업이 아닌 하나의 **개발 시스템**으로 설계한다.
CLAUDE.md(협업 규칙), Skills(반복 워크플로우), Hooks(자동 품질 게이트), Scheduled Tasks(정기 리포트), 코드 컨벤션이 유기적으로 연결된 체계를 구축하고, 문제가 발생하면 시스템의 해당 부분만 수정하여 전체 일관성을 유지한다.
이 시스템을 지속적으로 고도화하면서, AI와의 협업 품질 자체를 점진적으로 개선해 나간다.

## 2. 핵심 역량

### UX 중심 개발

최종 사용자 경험에 대한 감도가 높다. 기능 구현 후 직접 사용하며 반복적으로 개선하는 것이 일관된 패턴이다.

- 봇 응답 속도에 민감 → 자주 쓰는 패턴은 LLM을 거치지 않는 fast path로 직접 응답 속도를 단축
- "루틴 체크리스트가 대화에 밀려서 올라간다" → 별도 App Home 탭을 도입하여 UX 문제를 구조적으로 해결
- 버튼 위치, 액션 접근성, 정보 노출 우선순위를 직접 테스트하며 조정
- 일정 표시 포맷(카테고리 그룹, 상태 아이콘, 메모 └ 포맷), 모바일 safe-area, 드래그앤드롭 인터랙션까지 세밀하게 설계
- 대시보드 로딩 시간, DB cold start(1\~3초) 등 체감 성능을 인지하고 UX 관점에서 대응

### 풀스택 실행력

백엔드(Node.js, PostgreSQL, API 설계)와 프론트엔드(React, Next.js, CSS)를 동시에 설계·구현한다.

- Slack 봇 서버 + 웹 대시보드를 하나의 프로젝트로 구축
- Next.js API 라우트, 세션 인증, DB 쿼리부터 반응형 레이아웃, `calc(5rem + safe-area-inset-bottom)` 같은 디바이스 대응까지 전 영역을 다룸
- 인프라(Docker, CI/CD, Vercel 배포, Oracle VM)까지 직접 구축·운영

### AI 시스템 설계

LLM의 tool use 메커니즘, 프롬프트 엔지니어링, 에이전트 루프 패턴을 깊이 이해한다.

- "LLM이 요일을 머릿속으로 계산한다"는 문제를 정확히 진단하고 시스템적으로 해결
- 프롬프트 규칙의 강도를 상황에 따라 조절 (권장 → 절대 규칙)
- 타 프로젝트 아키텍처를 "시스템 프롬프트 + 도구 + 컨텍스트 주입"으로 정확히 분해하고 자기 프로젝트에 맞게 재설계

### 비용·성능 최적화

운영 비용과 성능을 동시에 고려하는 설계 습관이 있다.

- LLM 하이브리드 모델: 메인(Claude Sonnet) + 경량(Gemini Flash) 분리로 비용 절감
- fast path 패턴: 정규식 매칭 → SQL 직접 조회 → Block Kit 응답으로 LLM 호출 자체를 바이패스
- 인프라 비용 최적화: Oracle Free Tier(봇) + Vercel Free(웹) + Neon Free(DB)로 운영비 최소화

### 데이터 안전 우선

기능보다 데이터 안전을 우선한다.

- soft-delete를 자연스럽게 요구
- 마이그레이션 시 기존 데이터 호환성 확인
- SQL 도구에 DDL 차단, 위험 쿼리 방지 로직 설계

## 3. 문제 해결과 의사결정

### 원인 분석 우선

"이거 고쳐줘"보다 "이거 이상한데, 원인 찾아봐"가 먼저 나온다.
KST 타임존 버그 발견 시에도 "3월 28일이 토요일인데 금요일이라고 나온다. 원인 찾아봐"로 시작.
원인을 이해한 뒤에 수정을 판단하는 패턴이 일관적이다.

### 비교 기반 학습

새 기능 도입 시 유사 프로젝트를 연구하고, 차이점을 분석한 뒤 자기 프로젝트에 맞게 재설계한다.
남의 것을 그대로 가져오지 않고, 원리를 이해한 뒤 규모에 맞게 축소/변형한다.

### 엣지 케이스 선제 인식

구현 전에 실패 시나리오를 먼저 떠올린다.
스마트 메모리 설계 시 기능 요청과 동시에 "이러면 중요 프롬프트가 소실될 위험이 있나?"를 바로 질문.
이 질문이 soft-delete + source 이중 구조라는 핵심 설계를 이끌어냄.

### 회고 기반 방향 전환

v1 → v2 → v3로 이어지는 아키텍처 전환이 대표적.
"잘된 점 → 드러난 한계 → 결론"의 구조화된 회고를 통해 방향을 결정.
감정적 판단이 아닌 구조적 분석("단일 서버 한계 → 서비스별 분리")으로 전환한다.

## 4. 작업 방식

### 구조화된 프로세스

- 1 Issue = 1 PR = 1 Feature 원칙
- Conventional Commits를 한글로 사용하며, 커밋 메시지가 변경의 "왜"를 설명
- 커밋 포인트와 PR 경계를 인식하고, 주제가 섞이기 전에 끊어냄
- GitHub Issues로 개발 계획 관리, `docs/project-history.md`에 의사결정 과정 기록

### 빠른 실행 → 즉시 개선

기능을 먼저 구현한 뒤, 직접 사용하면서 빠진 부분을 발견하고 즉시 개선한다.
feat 커밋 직후 fix 커밋으로 UX를 다듬고, 기능 완성 직후 리팩토링으로 전환하는 "build → clean" 패턴이 일관적이다.
각 반복마다 명확한 방향성을 제시하며, 결과물을 확인한 뒤 요구사항을 정교하게 다듬어 나간다.

### AI 협업 시스템 운영

Claude Code의 기능을 조합해 반복 작업을 시스템화한다.

- 개발 리포트 생성·전송을 Scheduled Task로 자동화
- 코드 리뷰·컨벤션 점검을 Skill로 등록
- `.claude/settings.local.json`으로 AI 협업 환경 자체를 코드화
- 탐색/조사는 AI에 위임하되, 설계 결정과 최종 판단은 직접 수행

## 5. 성장 방향

### 요구사항 사전 정의 강화

기능을 반복적으로 다듬는 접근이 개인 프로젝트에서는 효과적이지만, 팀 환경에서는 초기에 최종 형태를 정의하고 역산하는 것이 협업 효율을 높일 수 있다. "최종 상태"를 먼저 그려보는 습관을 적용 중.

### 테스트 선행 전환

기능 구현 후 테스트를 보강하는 흐름에서, 핵심 로직은 테스트를 먼저 작성하고 구현하는 방식으로 전환을 시도하고 있다. 최근 vitest 기반 테스트 인프라를 구축하고 핵심 유틸리티·라우팅 로직의 테스트 커버리지를 확보한 것이 첫 단계.
