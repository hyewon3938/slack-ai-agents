# /start-feature — 새 기능 작업 시작

새 기능 개발의 전체 준비 과정을 자동화한다.
이슈 생성 → 브랜치 → 설계 → 컨텍스트 로드까지 한번에 처리.

## 입력

$ARGUMENTS

위 내용이 비어있으면 어떤 기능을 만들지 사용자에게 질문한다.

## 사전 로드 (자동)
아래 파일들을 읽어 프로젝트 맥락을 파악한다:
- `CLAUDE.md` — 프로젝트 규칙, 아키텍처
- `docs/conventions.md` — 코드 컨벤션, 라벨 체계, 리팩토링 기준
- `docs/project-history.md` — 지금까지의 의사결정 히스토리

## 기능 인터뷰

사전 로드가 끝나면, 사용자의 기능 설명을 바탕으로 **설계에 필요한 질문들을 인터뷰 형식으로 진행**한다.

### 인터뷰 규칙
- 기능 설명을 듣고, 설계·구현에 필요하지만 아직 불명확한 부분을 질문한다
- 한 번에 2\~3개씩 질문 (너무 많으면 부담)
- 질문 예시:
  - UX/동작: "이 기능이 트리거되는 시점은?", "결과가 어떤 형태로 보여야 해?"
  - 범위: "기존 기능과 어떻게 연결돼?", "MVP 범위는 어디까지?"
  - 기술: "DB 스키마 변경이 필요할 것 같은데 괜찮아?", "외부 API 연동이 필요해?"
  - 엣지케이스: "데이터가 없을 때는 어떻게 처리할까?", "에러 상황은?"
- 사용자 답변을 받으면 추가 질문이 필요한지 판단
  - 설계에 충분한 정보가 모이면 → "인터뷰 끝! 정리하면..." 으로 요약 후 다음 단계 진행
  - 아직 불명확하면 → 추가 질문 계속. 단, 매 라운드마다 "설계에 충분한 정보가 모였는지" 스스로 판단하고, 충분하다 싶으면 자연스럽게 마무리한다
  - 사용자가 "됐어", "이 정도면 충분해" 등으로 끊으면 즉시 마무리
- 소규모 수정이 명확한 경우(버그 수정, 값 변경 등) → 인터뷰 스킵

### 인터뷰 완료 시
- 확인된 내용을 간결하게 요약 (기능 목표, 범위, 주요 결정사항)
- 이 요약을 기반으로 규모 판단 및 설계 진행

## 규모 판단 (자동)

작업 시작 전, 요청의 규모를 판단한다:

**소규모 수정** (아래 중 하나 이상 해당):
- 커밋 1개로 끝나는 버그 수정, 스타일 수정, 설정 변경
- 기존 코드의 값/옵션만 바꾸는 수준
- 새 파일 생성이 없거나 1개 이하
- 설계 결정이 불필요한 명확한 작업

→ **이슈/브랜치 스킵**: main에 직접 커밋. 1\~3단계 건너뛰고 4단계(설계)부터 간략하게 진행.
→ 커밋 메시지에 `fix:` 또는 `feat:` 프리픽스만 사용.
→ 6\~7단계(코드 리뷰, PR)도 스킵.

**일반 기능** (위에 해당하지 않는 경우):
→ 이슈 생성 → 브랜치 → 설계 → PR 전체 플로우 진행.

**복수 항목 묶음** (여러 소규모 수정을 한번에 요청한 경우):
→ 이슈 1개에 체크리스트로 묶고, 브랜치 1개에서 작업. 커밋은 항목별 또는 통합.

판단이 애매하면 사용자에게 "이슈 만들까, 바로 작업할까?" 확인.

## 실행 단계

### 1단계: 기존 이슈 확인
- `gh issue list --state open`으로 열린 이슈 확인
- 관련 이슈가 이미 있으면 사용자에게 알려주고 해당 이슈를 사용할지 확인
- 없으면 새 이슈 생성으로 진행

### 2단계: 이슈 생성
- `docs/conventions.md`의 라벨 체계를 읽고 적절한 라벨 선택
- 기능 설명을 체크리스트 형태로 정리
- `gh issue create`로 이슈 생성 (제목, 본문, 라벨)
- 이슈 번호 기록

### 3단계: 브랜치 생성
- `docs/conventions.md`의 브랜치 전략에 맞는 브랜치명 생성
- `git checkout -b feature/이슈번호-기능명`
- 원격 push: `git push -u origin 브랜치명`

### 4단계: 기능 설계
- 현재 코드 구조를 분석하여 변경/추가할 파일 목록 작성
- DB 스키마 변경이 필요하면 마이그레이션 계획 포함
- 설계 방식의 장단점, 발생 가능한 리스크 설명
- "지금 필요한 건지, 오버엔지니어링인지" 판단 제시
- **🧪 TDD 대상 분류** — 설계 시 아래 기준으로 TDD 적용 여부를 명시:
  - 순수 함수 / 유틸리티 → **TDD 필수** (테스트 먼저 작성 → 구현)
  - DB 쿼리 로직 / 데이터 변환 → **TDD 권장**
  - API 라우트 핸들러 → **TDD 선택** (통합 테스트가 더 적합할 수 있음)
  - UI 컴포넌트 → **TDD 스킵** (시각적 확인이 더 효과적)
  - TDD 대상이 없으면 "TDD 대상 없음" 명시
- **⛔ 보안 영향도 분석** (아래 해당 시 반드시 포함):
  - 새 API 엔드포인트 → 인증 방식, 입력 검증 계획
  - DB 변경 → SQL 인젝션 방지 전략 (파라미터 바인딩, 컬럼 화이트리스트)
  - 인프라 변경 (Docker, 포트, nginx) → 외부 노출 범위, 시크릿 전달 방식
  - 외부 통신 추가 → HTTPS 여부, 토큰/키 관리 방식
  - 해당사항 없으면 "보안 영향 없음" 명시
- 사용자 확인 후 진행

### 5단계: 설계 확정 후
- 이슈에 설계 내용 코멘트 추가 (`gh issue comment`)
- `docs/project-history.md`에 마일스톤 기록:
  - 날짜, 기능명, 왜 이 기능을 만들었는지, 설계 결정 사항
- "구현을 시작하세요" 안내

### 작업 중 규칙
- **TDD 대상부터 먼저 개발**: 4단계에서 TDD로 분류된 항목은 테스트 먼저 작성 → 실패 확인 → 구현 → 통과 순서로 진행
- 커밋 시 이슈 번호 참조: `feat: 기능 설명 (#이슈번호)`
- 커밋이 3\~5개 쌓이면 PR 생성 제안
- 하나의 브랜치에 다른 기능이 섞이기 시작하면 PR 머지 후 새 브랜치 전환 권유
- **⛔ 커밋 전 보안 점검**: 코드/커밋에 시크릿, IP, 도메인 등 민감정보 포함 여부 확인

### 작업 완료 시

#### 6단계: 코드 리뷰 + 리팩토링
구현이 끝나면 `/review-code` 스킬의 리뷰 프로세스를 직접 실행한다:
- `docs/conventions.md` 기준으로 코드 품질, 컨벤션, 리팩토링 필요성 점검
- 🔴 필수 수정 사항은 즉시 수정
- 🟡 권장 사항은 사용자에게 수정 여부 확인
- 리팩토링 진행 시 같은 브랜치에서 커밋
- 리뷰 결과 요약을 사용자에게 보여준 뒤 다음 단계 진행

#### 7단계: 테스트 코드 작성
코드 리뷰 완료 후, 테스트 코드가 필요한지 판단한다:
- **작성 기준** (아래 중 하나 이상 해당하면 작성):
  - 순수 함수 / 유틸리티가 새로 추가됨
  - DB 쿼리 로직 / 데이터 변환이 복잡함
  - 엣지케이스가 많은 비즈니스 로직
  - 4단계에서 TDD로 분류되었지만 아직 테스트가 없는 항목
- **스킵 기준** (아래에 해당하면 생략):
  - UI 컴포넌트만 변경 (시각적 확인으로 충분)
  - 단순 CRUD API 라우트 (보일러플레이트)
  - 설정/환경변수 변경만
- 작성 시: 테스트 실행 → 통과 확인 → 커밋
- 스킵 시: "테스트 불필요 — [사유]" 명시

#### 8단계: PR 생성
- PR 생성 (`gh pr create`)
- 이슈 체크리스트 업데이트
- `docs/project-history.md` 업데이트 (결과, 배운 점)

## 주의사항
- 설계 없이 코드 작성을 시작하지 않는다.
- 모든 결정에 근거를 설명한다.
- PR/커밋에 민감한 env 정보 포함 금지.
