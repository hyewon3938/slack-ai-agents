# /design — 기능 설계 (Opus)

기능 요구사항 분석 → 인터뷰 → 설계 → 구현 계획서 작성까지.
Sonnet이 `/build`로 기계적으로 실행할 수 있을 만큼 디테일한 설계를 남긴다.

## 입력

$ARGUMENTS

위 내용이 비어있으면 어떤 기능을 만들지 사용자에게 질문한다.

## 사전 로드 (자동)
아래 파일들을 읽어 프로젝트 맥락을 파악한다:
- `CLAUDE.md` — 프로젝트 규칙, 아키텍처
- `docs/conventions.md` — 코드 컨벤션, 라벨 체계, 리팩토링 기준
- `docs/project-history.md` — 지금까지의 의사결정 히스토리

## 기능 인터뷰

사전 로드가 끝나면, 사용자의 기능 설명을 바탕으로 **설계에 필요한 질문들을 인터뷰 형식으로 진행**한다.

### 인터뷰 규칙
- 기능 설명을 듣고, 설계·구현에 필요하지만 아직 불명확한 부분을 질문한다
- 한 번에 2\~3개씩 질문 (너무 많으면 부담)
- 질문 예시:
  - UX/동작: "이 기능이 트리거되는 시점은?", "결과가 어떤 형태로 보여야 해?"
  - 범위: "기존 기능과 어떻게 연결돼?", "MVP 범위는 어디까지?"
  - 기술: "DB 스키마 변경이 필요할 것 같은데 괜찮아?", "외부 API 연동이 필요해?"
  - 엣지케이스: "데이터가 없을 때는 어떻게 처리할까?", "에러 상황은?"
- 사용자 답변을 받으면 추가 질문이 필요한지 판단
  - 설계에 충분한 정보가 모이면 → "인터뷰 끝! 정리하면..." 으로 요약 후 다음 단계 진행
  - 아직 불명확하면 → 추가 질문 계속
  - 사용자가 "됐어", "이 정도면 충분해" 등으로 끊으면 즉시 마무리
- 소규모 수정이 명확한 경우(버그 수정, 값 변경 등) → 인터뷰 스킵

### 인터뷰 완료 시
- 확인된 내용을 간결하게 요약 (기능 목표, 범위, 주요 결정사항)
- 이 요약을 기반으로 규모 판단 및 설계 진행

## 규모 판단 (자동)

작업 시작 전, 요청의 규모를 판단한다:

**소규모 수정** (아래 중 하나 이상 해당):
- 커밋 1개로 끝나는 버그 수정, 스타일 수정, 설정 변경
- 기존 코드의 값/옵션만 바꾸는 수준
- 새 파일 생성이 없거나 1개 이하
- 설계 결정이 불필요한 명확한 작업

→ **Opus가 직접 구현까지 처리**하고 끝. 이슈/브랜치/설계 파일 스킵.
→ main에 직접 커밋. `fix:` 또는 `feat:` 프리픽스만 사용.

**일반 기능** (위에 해당하지 않는 경우):
→ 이슈 생성 → 브랜치 → 설계 → 설계 파일 저장 → `/build`로 핸드오프.

**복수 항목 묶음** (여러 소규모 수정을 한번에 요청한 경우):
→ 이슈 1개에 체크리스트로 묶고, 브랜치 1개에서 작업.

판단이 애매하면 사용자에게 "이슈 만들까, 바로 작업할까?" 확인.

## 실행 단계

### 1단계: 기존 이슈 확인
- `gh issue list --state open`으로 열린 이슈 확인
- 관련 이슈가 이미 있으면 사용자에게 알려주고 해당 이슈를 사용할지 확인
- 없으면 새 이슈 생성으로 진행

### 2단계: 이슈 생성
- `docs/conventions.md`의 라벨 체계를 읽고 적절한 라벨 선택
- 기능 설명을 체크리스트 형태로 정리
- `gh issue create`로 이슈 생성 (제목, 본문, 라벨)
- 이슈 번호 기록

### 3단계: 브랜치 생성
- `docs/conventions.md`의 브랜치 전략에 맞는 브랜치명 생성
- `git checkout -b feature/이슈번호-기능명`
- 원격 push: `git push -u origin 브랜치명`

### 4단계: 코드베이스 분석 + 디테일 설계

현재 코드를 철저히 분석하고, **Sonnet이 판단 없이 그대로 실행할 수 있는 수준**의 구현 계획서를 작성한다.

#### 설계 아웃풋 필수 항목

```markdown
# 구현 계획: {기능명} (#{이슈번호})

## 개요
- 목표: [한 줄 요약]
- 관련 이슈: #{이슈번호}
- 브랜치: feature/{이슈번호}-{기능명}

## 파일별 변경사항

### 1. `경로/파일명.ts` (수정|신규|삭제)
- 변경 이유: [왜 이 파일을 수정하는지]
- Before (라인 XX\~YY):
  ```typescript
  // 기존 코드 (실제 코드 그대로)
  ```
- After:
  ```typescript
  // 변경할 코드 (실제로 넣을 코드 그대로)
  ```
- 주의사항: [있으면]

### 2. `경로/새파일.ts` (신규)
- 역할: [이 파일이 하는 일]
- 전체 구조:
  ```typescript
  // 파일 전체 스켈레톤
  // 함수 시그니처 + 핵심 로직 + 주석
  // import 포함
  ```

## DB 마이그레이션 (해당 시)
```sql
-- 정확한 마이그레이션 SQL
```
- 마이그레이션 파일명: `db/migrations/XXX_이름.sql`

## 테스트 케이스
| 함수 | 테스트명 | 입력 | 기대 출력 | 비고 |
|------|----------|------|-----------|------|
| detectX | 정상 감지 | {...} | {type: 'x'} | |
| detectX | 임계값 미달 | {...} | null | |

## 구현 순서 + 커밋 단위
1. [파일A, 파일B 수정] → 커밋: "feat: 기능 설명 (#이슈번호)"
2. [테스트 작성] → 커밋: "test: 테스트 설명 (#이슈번호)"
3. [프롬프트 수정] → 커밋 1에 포함 또는 별도

## TDD 대상 분류
- ✅ TDD 필수: [함수명] — 테스트 먼저 작성 → 구현
- ⬜ TDD 스킵: [사유]

## 보안 영향도
[분석 결과 or "보안 영향 없음"]

## 엣지케이스 + 주의사항
- [구현 시 주의할 점들]
```

#### 설계 품질 기준
- **코드 스니펫은 복붙 가능한 수준**: 타입, import, 변수명 모두 정확하게
- **Before/After를 명확하게**: 정확한 라인 범위, 기존 코드 원문 포함
- **신규 파일은 전체 구조**: import부터 export까지 스켈레톤 제공
- **테스트 케이스는 입출력까지**: 모킹 데이터 포함
- **커밋 단위 명확**: 어디서 끊을지 Sonnet이 고민 안 하게

### 5단계: 설계 확정

- 설계 계획서를 `.claude/plans/{이슈번호}-{기능명}.md`로 저장
- 이슈에 설계 내용 코멘트 추가 (`gh issue comment`)
- `docs/project-history.md`에 마일스톤 기록:
  - 날짜, 기능명, 왜 이 기능을 만들었는지, 설계 결정 사항

### 6단계: 핸드오프 안내

아래 메시지를 출력한다:

```
설계 완료! 구현으로 넘어가려면:

1. `/compact` — 설계 과정 컨텍스트 정리 (토큰 절약)
2. `/build` — 구현 시작

설계 파일: .claude/plans/{이슈번호}-{기능명}.md
```

> `/compact`를 먼저 실행하면 설계 탐색 과정이 요약되어 `/build` 단계에서 불필요한 입력 토큰을 줄일 수 있다.
> model frontmatter가 자동 전환하지 않는 경우, `/compact` 후 `/model sonnet`을 추가로 입력한다.

## 주의사항
- 설계 없이 코드 작성을 시작하지 않는다 (소규모 수정 제외).
- 모든 결정에 근거를 설명한다.
- PR/커밋에 민감한 env 정보 포함 금지.
- 설계의 디테일 수준이 이 스킬의 핵심 가치. **"Sonnet이 읽고 바로 구현 가능한가?"**를 자문하며 작성.
