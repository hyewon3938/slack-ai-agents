# /review-code — 코드 리뷰 + 컨벤션 점검

현재 브랜치의 변경사항을 리뷰하고, 컨벤션 준수 여부를 점검한다.
리팩토링이 필요하면 제안하고, 컨벤션 자체의 업데이트도 검토한다.

## 입력

$ARGUMENTS

비어있으면 현재 브랜치의 모든 변경사항을 리뷰한다.
특정 파일이나 범위를 지정할 수도 있다.

## 사전 로드 (자동)
- `docs/conventions.md` — 코드 컨벤션 + 리팩토링 기준
- `CLAUDE.md` — 프로젝트 규칙

## 실행 단계

### 1단계: 변경사항 파악
- `git diff main...HEAD`로 현재 브랜치의 전체 변경 확인
- 변경된 파일 목록, 추가/삭제 라인 수 요약

### 2단계: ⛔ 보안 감사 (최우선)
> Public 저장소 + 개인 데이터 프로젝트이므로 보안을 가장 먼저 점검한다.
> `docs/conventions.md`의 보안 체크리스트를 반드시 참조한다.

**시크릿 유출 점검**:
- 코드/커밋에 API 키, 비밀번호, 토큰, IP, 도메인 등 하드코딩 여부
- .env.example과 docker-compose.yml 환경변수 동기화 여부
- 커밋 히스토리에 민감정보 유입 여부 (`git log -p`로 확인)

**API/웹 보안 점검** (해당되는 경우):
- 새/변경된 API 라우트에 인증 검증이 있는가?
- 사용자 입력의 서버 측 검증 (타입, 범위, 화이트리스트)
- SQL 쿼리에 파라미터 바인딩 사용 여부 (문자열 결합 금지)
- 에러 응답에 내부 구현 정보 노출 여부

**인프라 보안 점검** (해당되는 경우):
- Docker/nginx/포트 변경 시 외부 노출 범위 확인
- 쿠키/세션 설정 (HttpOnly, SameSite, Secure) 적절성
- CORS 정책, 보안 헤더 설정

**보안 이슈는 무조건 🔴 (필수 수정)**. "나중에 고치자" 불가.

### 3단계: 코드 품질 리뷰
각 변경 파일에 대해 검토:
- **버그 위험**: 잠재적 버그, edge case 누락, 에러 핸들링 부족
- **성능**: 불필요한 연산, N+1 쿼리, 메모리 누수 가능성
- **가독성**: 네이밍, 함수 길이, 복잡도
- **테스트**: 테스트 커버리지, 엣지 케이스 테스트 여부

심각도별 분류: 🔴 필수 수정 / 🟡 권장 / 🟢 참고

### 4단계: 컨벤션 준수 점검
`docs/conventions.md`의 규칙과 실제 코드를 비교:
- 네이밍 컨벤션 준수 여부
- 파일 구조 규칙 준수 여부
- 커밋 메시지 컨벤션 준수 여부
- import 정렬, 에러 핸들링 패턴 등

### 5단계: 컨벤션 업데이트 제안
실제 코드 패턴과 컨벤션이 다른 경우:
- 새로운 패턴이 반복적으로 사용되고 있으면 → 컨벤션에 추가 제안
- 기존 규칙이 현재 프로젝트에 맞지 않으면 → 규칙 수정 제안
- 사용자 승인 시 `docs/conventions.md` 직접 업데이트

### 6단계: 리팩토링 제안
`docs/conventions.md`의 리팩토링 기준에 따라:
- 리팩토링이 필요한 코드 식별
- 각 항목에 대해 "지금 해야 할 것 vs 나중에 해도 될 것" 분류
- 사용자 승인 시 리팩토링 코드 작성

### 7단계: 결과 요약
- 전체 리뷰 요약 (파일 수, 이슈 수, 심각도 분포)
- 필수 수정 사항 체크리스트
- 컨벤션 업데이트 사항 (있으면)
- 리팩토링 제안 목록 (있으면)

## 주의사항
- 프로젝트의 큰 틀(아키텍처, 설계 방향)을 벗어나는 리팩토링은 제안하지 않는다.
- 사소한 스타일 이슈보다 실질적 품질 문제에 집중한다.
- 리팩토링은 제안만 하고, 실행은 사용자 승인 후에 한다.
